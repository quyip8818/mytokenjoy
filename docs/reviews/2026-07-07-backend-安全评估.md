# Backend 安全评估报告

- **评估对象**：`apps/backend`（Go 1.24 + chi + pgx v5 / PostgreSQL + NewAPI）
- **评估日期**：2026-07-07
- **评估范围**：认证与端点覆盖面、RBAC 越权（横向/纵向）、敏感数据处理、注入类风险、CORS/限流等横切项
- **方式**：只读代码审计
- **后续**：原 C1（跨租户 ConfirmRecharge）已于后续修复（`ConfirmPayment` 校验 `order.CompanyID == company.CompanyID(ctx)`）。

---

## 一、结论摘要

整体认证与租户隔离链路（JWT session → `CompanyResolve` 注入 company → `RequireSession` 校验 → store 按 `CompanyID(ctx)` 过滤）设计合理，PostgreSQL 层参数化良好、未发现经典 SQL 注入或 RCE。主要风险集中在**上游/平台 Key 明文存储与过度返回**，以及 **gateway 的策略边界**。

按严重度统计：

| 严重度   | 数量 | 关键问题                                                                                         |
| -------- | ---- | ------------------------------------------------------------------------------------------------ |
| Critical | 2    | Provider/Platform Key 明文入库                                                                     |
| High     | 6    | Sync Key 绕过鉴权；角色提升；审计设置权限错配；平台 token secret 回退；Platform Key 持续返回明文；demo 默认加密密钥 |
| Medium   | 8    | 冒充他人审批、内部 metrics 无认证、错误信息泄露、gateway 路径/白名单、ILIKE 通配符等             |
| Low      | 若干 | 时序枚举、常量时间比较、defense-in-depth 缺口                                                    |

## 二、Critical

### C2. Provider Key（上游 LLM 密钥）明文入库

- **文件**：`internal/domain/keys/provider_key.go:39,112-113`；`internal/store/postgres/keys_repo.go:20-21,34,72-79,86`；`internal/store/postgres/schema.sql:298`
- **问题**：创建/轮换时 `SecretKey` 直接写入 `provider_keys.secret_key` 明文列，无 hash、无加密。DB 泄露即全部上游密钥泄露。（API 层有 `json:"-"` 阻止响应暴露，但存储层不合规。）
- **修复方向**：改为 KMS/应用层加密存储，或最小化存储范围。

### C3. Platform Key（Platform Key）明文入库

- **文件**：`internal/domain/keys/platform_key_newapi.go:27-36,51`；`internal/store/postgres/keys_repo.go:97-98,111,167-173,181`；`internal/store/postgres/schema.sql:313`
- **问题**：NewAPISync 同步后将完整 token 存入 `platform_keys.full_key` 明文列，Gateway 鉴权也依赖该列等值查询（`newapi.go:70-75`）。
- **修复方向**：存储 hash + 前缀，Gateway 鉴权改为按 hash 查找。

## 三、High

### H1. Sync API Key 完全绕过 session 与权限

- **文件**：`internal/http/middleware/sync_trigger.go:19-22`
- **问题**：当 `X-Sync-API-Key` 匹配配置值时，直接放行，跳过 `RequireSession` 与 `org:datasource` 权限。
- **攻击场景**：密钥泄露即可未授权触发 `POST /api/org/sync/trigger`，篡改默认 tenant 组织数据。
- **修复方向**：即便走 API Key 也应绑定 company + 记审计日志，不完全 bypass authz。

### H2. `UpdateMember` 无角色提升防护

- **文件**：`internal/domain/org/structure/member.go:95-118`
- **问题**：`UpdateMember` 将请求体 `input.Roles` 原样写入，未校验是否允许分配 `超级管理员` 等 preset 角色。
- **攻击场景**：拥有 `org:members` 的用户通过 `PUT /api/org/members/{id}` 把自己或同伙的 roles 设为 `["超级管理员"]`，获得全部权限。
- **修复方向**：禁止通过成员 body 直接改 `Roles`（改角色应走 `org:roles` 的角色成员 API），并 block preset 超管角色名。

### H3. 审计设置写操作仅需 `audit:read`

- **文件**：`internal/http/handler/audit/handler.go:84-86`；`internal/infra/permission/manifest.json:57-66`
- **问题**：`PUT /api/audit/settings` 只检查 `audit:read`，`audit:read` 不属于 write capabilities，只读审计员即可修改审计保留策略。
- **修复方向**：新增/复用 write 权限（如 `audit:write`）。

### H4. 平台 token secret 回退 + Bearer 共用导致越权

- **文件**：`internal/identity/httpx/token.go:44-55,136-145`；`internal/config/config.go:127-131`
- **问题**：`ResolvedPlatformSessionSecret()` 在 `PLATFORM_SESSION_SECRET` 为空时回退到 `SESSION_SECRET`；member 与 platform 均从 `Authorization: Bearer` 读取 token。
- **攻击场景**：运维未设 `PLATFORM_SESSION_SECRET` 且开启 SaaS，普通 member 用同一 JWT 调用 `POST /api/platform/companies/{id}/recharge` 等平台接口。
- **修复方向**：platform 解析仅接受 platform cookie，或禁止 secret 回退。

### H5. Platform Key 的 `fullKey` 在 list/update/toggle 持续返回

- **文件**：`internal/domain/types/keys.go:20-21`；`internal/domain/keys/platform_key_list.go:40-46`、`platform_key_enrich.go:34-59`、`platform_key_update.go:103`、`platform_key_actions.go:34-38`；`internal/http/handler/keys/handler.go:102-110,125,135,145`
- **问题**：`GET /api/keys/platform` 及 update/toggle 响应可能持续携带完整 `fullKey`，违反"仅创建时返回一次"约定。任何 `keys:read` 管理员可批量导出全部有效 key。
- **修复方向**：list/update 响应剥离 `fullKey`，仅 create 返回一次。

### H6. Demo 模式数据源凭证使用硬编码默认加密密钥

- **文件**：`internal/domain/org/core/credentials.go`；`internal/pkg/common/crypto.go:33-35`（`DevDefaultKey()`）
- **问题**：未设置 `DATA_SOURCE_CREDENTIAL_KEY` 时，飞书/钉钉/企微凭证曾用可预测的固定 32 字节密钥加密。
- **状态**：**已修复** — `Load()` 强制校验 `DATA_SOURCE_CREDENTIAL_KEY`；`CredentialKey()` 不再调用 `DevDefaultKey()`。

> 另有配置/文档类弱 secret：`Makefile` 默认 `SESSION_SECRET=tokenjoy-dev-session-secret`、`.env.example` 占位 secret；`config.go` 仅校验非空不校验强度。

## 五、Low

- **L1** `internal/identity/credentials/service.go:48-61`：用户不存在时跳过 bcrypt，存在用户时才做，存在时序侧信道可枚举有效邮箱（文案已统一为 `Invalid credentials`）。
- **L2** `internal/http/middleware/sync_trigger.go:19`、`internal/http/handler/ingest/handler.go`：secret 比较未用 `crypto/subtle.ConstantTimeCompare`。
- **L3** `internal/http/handler/session/handler.go:23-39`：`/api/session` 未走 `RequireSession` 中间件（手动解析，行为基本一致）。
- **L4** `internal/domain/gateway/gateway_service.go:124-134`：反向代理未显式 `path.Clean` + 前缀校验，且默认转发客户端原始 header（除 Authorization），缺少 defense-in-depth。
- **L5** `internal/store/postgres/sqlutil.go`、`newapi.go:111-115`：存在按字符串拼接表名/WHERE 的内部函数，当前调用点均为硬编码常量，安全，但易被误用需防回归。

## 六、已检查无重大问题的方面

- **密码存储**：邀请接受、平台 operator、seed 均用 `bcrypt.GenerateFromPassword`，验证用 `bcrypt.CompareHashAndPassword`（恒定时间），DB 仅存 hash。
- **日志泄露**：无 access log 记录请求体/头；Recover 仅记录 panic 值与 path；Gateway 读取的 Authorization 未写日志。
- **主 REST 错误响应**：`httputil.WriteError` 对非 `DomainError` 统一返回 `Internal server error`，不暴露内部细节。
- **SQL 注入**：postgres/usagequery 层参数化良好，ORDER BY 为静态字面量，IN 列表用 `= ANY($N)`。
- **SSRF/命令注入/目录穿越**：newapi/feishu 客户端 URL 来自配置、path 硬编码；运行时无 `os/exec`；无用户可控文件路径拼接。
- **平台/成员 token 隔离**：正常配置下 secret 分离，platform token（CompanyID=0）无法通过 `ParseMemberToken`。
