# 低余额严格预检方案

## 问题

当前 Gateway 预检链路：

```
请求 → LoadPrecheckContext (1次PG读) → Evaluate → softBudgetCheck (缓存)
```

`wallet_remain` 和 `gateway_soft_remain` 都是**最终一致**的：
- `wallet_remain` 由 ingest 事务更新，延迟 = ingest 入队 + 处理时间（通常 1-5s）
- `gateway_soft_remain` 由 Projector 批处理更新，延迟 = ingest + projection batch interval（通常 5-30s）

高余额时这个延迟无所谓——最坏情况多放行几个请求，overdraft 兜底。  
但余额接近 0 时，延迟窗口内可能放行大量请求，导致 overdraft 数额不可控。

## 核心思路

**余额高时乐观放行（当前逻辑不变），余额低时加一次实时校验。**

不做预扣，不做 FOR UPDATE，只多一次 SELECT 读取最新 `wallet_remain`。

## 数据时效性分析

```
wallet_remain 延迟 ≈ ingest 处理延迟（1-5s）
gateway_soft_remain 延迟 ≈ ingest + projector batch（5-30s）
```

低余额时的风险窗口 = 延迟时间 × 并发请求速率 × 平均单次消耗。

例：延迟 3s，10 req/s，均消 500 quota → 窗口内放行 15000 quota。

如果 wallet_remain 已经 < 15000，就应该走严格路径。

## 方案

### 改动范围

只改 `domain/gateway/precheck.go` 的 `PrecheckService.Run`，加一个条件分支。

### 逻辑

```go
func (p *PrecheckService) Run(ctx, keyHash, model, opts) error {
    row, err := p.loader.LoadPrecheckContext(ctx, keyHash)
    // ... 现有检查 ...

    // 低余额严格检查：穿透读一次最新 wallet_remain
    if row.WalletRemain < p.lowBalanceThreshold(row.CompanyID) {
        fresh, err := p.loader.LoadFreshWalletRemain(ctx, row.CompanyID)
        if err == nil && fresh < minEstimatePoint {
            return fmt.Errorf("insufficient wallet quota")
        }
    }

    return p.softBudgetCheck(ctx, ...)
}
```

### `LoadFreshWalletRemain` 实现

```sql
SELECT wallet_remain FROM companies WHERE id = $1
```

一次普通 SELECT，无锁。读到的是最近一次 ingest commit 后的值。  
延迟 < 1ms（索引命中主键）。

### 阈值

```go
const defaultLowBalanceThreshold = 50000 // quota，约等于 10 次 GPT-4 请求
```

后续可以按公司 TPS 动态调整，但先硬编码一个保守值够用。

### 为什么不用 FOR SHARE / FOR UPDATE

- `FOR SHARE` 虽然不阻塞写，但持有锁会影响 PG autovacuum 和 connection 状态
- 一次普通 SELECT 已经读到 committed 数据（PG 默认 Read Committed），足够准确
- 对于预检这种读多写少的场景，无锁读是正确选择

### 为什么 wallet_remain 的延迟可接受

`wallet_remain` 在 ingest 事务内更新（`ConsumeLots` → `SetWalletRemain`）。  
事务 commit 后，下一次 SELECT 就能读到新值。

最大延迟 = 一个 ingest 事务的执行时间（通常 < 100ms）。  
这和 "请求处理延迟" 是同一个量级——可以接受。

真正的延迟来源是 **入队到被处理**（1-5s），这段时间 wallet_remain 确实是旧的。  
但这种情况和"不做严格检查"比，已经从 30s 延迟（等 projector）缩短到 1-5s 延迟。

### 高并发下仍可能超额的场景

10 个请求同时到达，都读到 wallet_remain = 5000（> minEstimate），全部放行。  
但实际只够 1 个请求。

这种并发窗口是无法用纯读操作消除的——只有预扣（CAS / atomic decrement）才能消除。  
但代价是每个预检变成写操作，吞吐从 10k+ 降到 1k 以下。

**结论**：接受这个窗口（overdraft 兜底），不做预扣。

## 不做的事

| 方案 | 为什么不做 |
|------|-----------|
| 预扣（atomic decrement wallet） | 预检变写操作，吞吐暴跌 |
| Redis 做实时余额 | 多一层一致性问题，当前规模不需要 |
| 限流低余额公司 | 产品上不可接受——用户买了额度凭什么被限速 |
| 估算请求成本做精确比对 | 流式请求在完成前无法知道实际 token 数，估算不准 |

## 实施

1 个文件改动 + 1 个 SQL（加到 `GatewayPrecheckRepository`）。  
半天工作量，无 schema 变更，无新依赖。

## 收益

- 余额充足时：零影响，走现有快路径
- 余额偏低时：多一次 PG 主键读（< 1ms），将超额窗口从 ~30s 缩短到 ~3s
- 完全透支后：直接拒绝（已有逻辑）
