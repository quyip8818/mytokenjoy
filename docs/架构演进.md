# 架构演进

> **定位**：TokenJoy Backend 的**目标架构**与**长期维护必改项**。  
> **读者**：架构 / Backend 开发。  
> **关联**：[Backend-架构.md](./Backend-架构.md) · [Backend-预算.md](./Backend-预算.md) · [Backend-离线任务.md](./Backend-离线任务.md)

---

## 一、现状评价

底座方向正确——控制面/数据面分离、三 SSOT（`usage_ledger` / `company_recharge_lots` / `budget_consumed`）、modular monolith、ledger + 投影。不必推翻。

长期维护的瓶颈是：**事实能否重建、schema 能否演进、域边界是否在表上对齐、PlatformKey 与 NewAPI 双写铁律是否统一**。

---

## 二、终态设计原则

| # | 原则 | 含义 |
| --- | --- | --- |
| P0 | 摘要列是投影，不是事实 | `gateway_soft_*` 可由 ledger + 预算树重算 |
| P1 | 热路径零远程、零重算 | Gateway：1× PG + 0~1× Redis + 纯内存 Evaluate |
| P2 | 事实单写 | 消耗只追加 `usage_ledger`；钱包只经 `billing/lot/` |
| P3 | 投影可丢弃可重算 | `budget_consumed` / `usage_buckets` / 摘要列均可从 ledger 重建 |
| P4 | 冷路径合并同类项 | 同一 NewAPI 对齐一条 worker 链 |
| P5 | 执法分层、职责单一 | 钱包 hard → 摘要 soft → Overrun Disable |

### Gateway 量化指标（终态）

| 指标 | 要求 |
| --- | --- |
| Store 调用 | ≤ 1（`LoadPrecheckContext`） |
| 预检 HTTP | 0 |
| 钱包余额 | O(1) 读 `wallet_remain_quota` |
| 预算 | 读 Key 行摘要列，不 JOIN `budget_consumed` |

## 三、一致性模型

| 数据 | 一致性 | Gateway 可当事实？ | 漂移修复 |
| --- | --- | --- | --- |
| `wallet_remain_quota` | Strong | 是（L0 hard） | lot；`ReconcileWalletDrift` |
| `usage_ledger` | Strong（append-only） | 是（SSOT） | 不可改 |
| `budget_consumed` | Eventually consistent | 否 | `budget_reconcile` |
| `gateway_soft_*` / version | Eventually consistent | 是（L1，带 lag） | Projector / Reconcile |
| Redis soft | Best effort | 可选增强；miss 放行 | Projector / Reconcile |

## 四、落地优先级

| 优先级 | 内容 | 类型 |
| --- | --- | --- |
| **D1** | `platform_keys` 摘要三列 + version（schema） | Schema |
| D2 | `LoadPrecheckContext` 读摘要列 | 读路径 |
| C1 | Projector 批末 UPDATE 摘要 + version++ | 写路径 |
| C2 | Reconcile 修漂移后刷新摘要 | 写路径 |
| C3 | Evaluate 读 `gateway_soft_remain`；NULL 放行 | Gateway |
| C4 | Redis SET 带 version | Redis |
| C5 | Redis refresh 读摘要列 | 性能 |
| C8 | 监控 + Projection SLA 告警 | 可观测 |

**Schema 增量（D1）：**
```sql
ALTER TABLE platform_keys
  ADD COLUMN gateway_soft_remain  NUMERIC(18, 6),
  ADD COLUMN gateway_soft_at      TIMESTAMPTZ,
  ADD COLUMN gateway_soft_version BIGINT NOT NULL DEFAULT 0;
```

## 六、Projection SLA

| 级别 | `projection_lag_seconds` | 动作 |
| --- | --- | --- |
| 正常 | < 0.5s | — |
| warning | ≥ 0.5s | 日志 |
| degraded | ≥ 1s | 告警 |
| critical | ≥ 5s | 暂停大 tenant / 加 worker |

设计目标：P99 lag ≤ 1s。

## 七、长期必改的架构问题

### P0-1 · 派生层需「从事实重建」能力

Ingest 后 `budget.Projector` 异步写 consumed，reconcile 定期修漂移。但缺乏产品化的**闭环重建入口**。

**目标：** 任一租户可触发「从 ledger 重算某月 consumed / buckets」且 Gateway 行为不变。

### P0-2 · Schema 演进策略

当前 `go:embed schema.sql` 全量 apply + `down -v` 重建。有租户数据后无法 wipe。

**目标：** 引入 migration 目录（`store/postgres/migrations/`），启动时 version 检查；禁止 production wipe。

### P0-3 · 预检一致性模型未产品化

Gateway 热路径仅读 PG（wallet_remain_quota + gateway_soft_*），不参与 NewAPI quota。但无统一 Convergence 抽象。

**目标：** 统一 `DriftReport(company_id)` 漂移报告 + Gateway / Ingest 共享 Remain 计算。

### P1-1 · `org_nodes` 多域共表

组织树 + 预算配置 + 模型路由同一行。功能增长会列膨胀、锁耦合。

**目标：** 路由列迁出 → `org_node_routing`；预算配置迁出 → `org_node_budget`。

### P1-2 · PlatformKey 双写顺序不统一

Create/Revoke = Remote-first；Update = DB-first。失败模式不同。

**目标：** 统一为一种铁律（推荐 Remote-first + outbox）。

### P1-3 · 双轨账期复杂度

`OpenBudgetPeriod` vs `OccurrenceMonth` 仅在 Ingest 处理。新写账路径易出错。

**目标：** 类型层强制区分 + 所有写账路径统一入口。

### P2-1 · 执行面强绑定 NewAPI

数据面假设 NewAPI log 形状与 webhook 协议。

**目标：** 定义 `LogSource` 接口；NewAPI 为一种 adapter。替换执行面 = 换 adapter。

### P2-2 · `models` 平台与租户同表

同表靠 `company_id` 约定区分。平台「下架旧模型」影响租户历史。

**目标：** 逻辑拆表或加 `origin` + catalog 分层。

---

## 八、演进路线

```text
P0: migration 体系 → ledger 重算投影 → Convergence 漂移报告
P1: PlatformKey 双写统一 → org_nodes 拆表 → 双轨类型护栏
P2: LogSource 端口 → models catalog 分层
```

| 阶段 | 完成标志 |
| --- | --- |
| P0 | 生产可 migration；任一租户可触发重算且 Gateway 不变 |
| P1 | org_nodes 仅组织树；Key 只有一条 outbox 铁律 |
| P2 | 换执行面 adapter 不改 ledger；模型下架不影响审计 |

---

## 九、明确不做

Gateway JOIN `budget_consumed` · Redis 钱包 SSOT · Redis miss 回源 PG · Ingest 入队 overrun · Gateway 503 sync lag · DB migration 框架（用 golang-migrate / Atlas） · 未压测上 PrecheckContext cache · Projector 插件/Event 框架 · 多 Region/Kafka/CQRS · 拆微服务
