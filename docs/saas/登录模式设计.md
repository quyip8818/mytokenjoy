# 登录模式设计

TokenJoy 支持私有化和 SaaS 两种部署形态。**登录体验统一**，通过配置切换主认证方式。

---

## 1. 认证策略

两条对等的主认证路径，由 `AUTH_PRIMARY` 配置切换：

| 策略 | 适用市场 | 登录方式 | 注册方式 |
|------|---------|---------|---------|
| `phone` | 中国 | 手机号 + 短信验证码 | 手机号验证 → 创建企业 |
| `email` | 海外 | 邮箱 + 密码 或 邮箱 + 邮件验证码 | 邮箱验证 → 创建企业 |

**不是主/备关系**——两条路径代码对等、能力对等，通过配置决定哪条是默认展示。

## 3. 邮箱认证：两种模式

邮箱登录支持**密码**和 **OTP** 两种方式，用户可选：

| 模式 | 体验 | 适用场景 |
|------|------|---------|
| 邮箱 + 密码 | 传统，适合高频登录 | 已有密码的用户、私有化 |
| 邮箱 + OTP | 无密码，验证码发到邮箱 | 首次登录、忘记密码、安全敏感 |

## 4. 统一 OTP 服务

手机号验证码和邮箱验证码共用一套后端逻辑：

```go
type OTPService interface {
    Send(ctx context.Context, target string, channel Channel) error
    Verify(ctx context.Context, target string, code string) (bool, error)
}

type Channel string
const (
    ChannelSMS   Channel = "sms"
    ChannelEmail Channel = "email"
)
```

### API 端点

| 端点 | Body | 说明 |
|------|------|------|
| `POST /auth/otp/send` | `{ target, channel }` | 发送验证码到手机或邮箱 |
| `POST /auth/otp/verify` | `{ target, code }` | 验证 → 返回登录/注册引导 |
| `POST /auth/login` | `{ email, password, companyId? }` | 邮箱密码登录 |
| `POST /auth/register` | `{ companyName, target, token }` | 创建 Trial 企业 |

`/auth/otp/send` 根据 `channel` 选择投递方式：
- `sms` → 短信服务商（阿里云/腾讯云）
- `email` → SMTP 发送验证码邮件

## 5. 私有化 vs SaaS 差异

| 维度 | 私有化 | SaaS |
|------|--------|------|
| 主认证 | 由 `AUTH_PRIMARY` 决定（同 SaaS） | 由 `AUTH_PRIMARY` 决定 |
| 自助注册 | ❌（"未找到账号，请联系管理员"） | ✅（新用户自动创建 Trial） |
| 登录按钮 | "登录" | "登录 / 注册" |
| 企业数量 | 1 家（`LocalCompanyID`） | 多家（JWT `company_id`） |
| Trial | 不适用 | 注册即 Trial |
| 供应商 Key | 企业管理员完全 CRUD | 只读展示（"由平台运营管理"） |
| 充值 | 正常自助充值 | Trial 时自动弹充值引导，充值即升级为正式版 |

**代码路径差异**：
- `registrationEnabled` → 新用户报错 vs 走注册
- `companyType === 'trial'` → 充值流程变为升级确认
- `supportSaas` → 供应商 Key 页面只读

## 6. 功能行为差异

### 6.1 供应商 Key（SaaS 只读）

SaaS 模式下，供应商 Key 由平台运营统一管理。企业管理员看到的是只读视图：

```
┌──────────────────────────────────────────────┐
│  供应商 Key                                   │
│                                               │
│  ┌─ Info Banner ─────────────────────────┐   │
│  │ 供应商 Key 由平台统一配置和管理        │   │
│  └───────────────────────────────────────┘   │
│                                               │
│  ┌────────┬──────────┬────────┬───────┐      │
│  │ 名称   │ 供应商   │ Key 前缀│ 状态  │      │
│  │ GPT-4  │ OpenAI   │ sk-abc… │ 正常  │      │
│  │ Claude │ Anthropic│ sk-xyz… │ 正常  │      │
│  └────────┴──────────┴────────┴───────┘      │
│                                               │
│  （无新增/编辑/删除按钮）                     │
└──────────────────────────────────────────────┘
```

前端判断：`VITE_SUPPORT_SAAS=true` → 隐藏 CRUD 按钮，展示 Info Banner。

### 6.2 充值与 Trial 升级

Trial 公司的充值流程变为**升级确认**：

```
用户点击充值（或系统自动弹出充值引导）
    │
    ▼
┌────────────────────────────────────────┐
│  升级为正式版                           │
│                                        │
│  充值后您的企业将升级为正式版：         │
│  · 接入真实 AI 模型                    │
│  · 模拟资金清零，使用真实余额           │
│  · 所有数据保留                         │
│                                        │
│  充值金额: [________] 元               │
│                                        │
│  [取消]         [确认充值并升级]        │
│                                        │
└────────────────────────────────────────┘
```

确认后：
1. 创建充值订单（paid lot）
2. 执行 `ExpireMockLots`（模拟资金清零）
3. `companies.type` 改为 `standard`
4. Banner 消失，Key allowlist 解锁

---

前端启动时调用，决定登录页渲染：

```typescript
interface AuthCapabilities {
  primaryAuth: 'phone' | 'email'
  emailPasswordEnabled: boolean     // 是否允许邮箱+密码
  emailOtpEnabled: boolean          // 是否允许邮箱+OTP
  phoneOtpEnabled: boolean          // 是否允许手机+OTP
  registrationEnabled: boolean      // 是否允许自助注册
  supportSaas: boolean
}
```

## 10. 配置

| 环境变量 | 默认值 | 说明 |
|------|--------|------|
| `AUTH_PRIMARY` | `phone` | 主认证方式：`phone` 或 `email` |
| `SUPPORT_SAAS` | `false` | SaaS 多租户模式 |
| `REGISTRATION_ENABLED` | `true` | 允许自助注册（SaaS） |
| `SMS_PROVIDER` | — | 短信服务商（`aliyun` / `tencent`） |
| `SMTP_HOST` | — | 邮件服务（用于邮箱 OTP） |

前端：
| 变量 | 说明 |
|------|------|
| `VITE_SUPPORT_SAAS` | 与后端 `SUPPORT_SAAS` 对齐，控制前端 SaaS 行为 |

> `VITE_SUPPORT_SAAS` 由启动脚本自动设置，前端据此控制：
> - 登录按钮文案（"登录" vs "登录/注册"）
> - 供应商 Key 页面只读
> - Trial 充值升级流程

---

## 11. OTP 规则

| 配置 | 值 |
|------|---|
| 验证码长度 | 6 位数字 |
| 有效期 | 5 分钟（Redis TTL） |
| 发送间隔 | 60 秒 |
| 每日上限 | 10 次/target |
| 验证尝试 | 最多 5 次，超过锁定 15 分钟 |

手机和邮箱共用同一套规则，存储结构相同（key = `otp:{target}`）。

## 12. 实施优先级

| 阶段 | 内容 |
|------|------|
| **P0** | `GET /auth/capabilities` + 前端条件渲染 |
| **P1** | 统一 OTP 服务（Send/Verify）+ Redis 存储 + 防刷 |
| **P2** | 手机号 OTP 投递（SMS provider） |
| **P3** | 邮箱 OTP 投递（SMTP） |
| **P4** | `/auth/otp/verify` → 登录/注册合一逻辑 |
| **P5** | `/auth/register` 创建 Trial |
| **P6** | 多企业选择页 |

---

## 13. 设计原则

| 原则 | 说明 |
|------|------|
| 两条路径对等 | phone 和 email 不是主/备，是地区适配 |
| 配置切换不改代码 | `AUTH_PRIMARY` 改一个环境变量即切换 |
| OTP 统一抽象 | 手机和邮箱共用 Send/Verify 接口 |
| 私有化/SaaS 差异最小化 | 唯一 if：`registrationEnabled` |
| 邮箱双模式 | 密码 + OTP 并存，用户可选 |
