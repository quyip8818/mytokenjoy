# 用户身份解析与企业绑定

> 解决核心问题：用户登录时，系统如何定位到正确的企业（Company）？
> 覆盖两种场景：成员受邀加入已有企业 / 新用户自助注册新企业。

---

## 1. 问题定义

TokenJoy 是多租户系统，JWT 必须携带 `company_id`。但用户登录时只提供凭证（手机号/邮箱），不知道自己的 company_id。

| 场景 | 问题 |
|------|------|
| 私有化（单企业） | 无此问题——`LocalCompanyID` 隐式注入 |
| SaaS 受邀成员 | 该手机号/邮箱已存在于某企业的 members 表 → 需查找 |
| SaaS 新用户注册 | 手机号/邮箱不在任何企业 → 创建新 Trial 企业 |
| SaaS 多企业成员 | 同一手机号属于多个企业 → 需用户选择 |

---

## 2. 行业做法

| 产品 | 方式 |
|------|------|
| **飞书** | 手机号验证后，后端查找关联组织，多个时弹选择页 |
| **Slack** | 用户先输入 workspace URL 定位企业，再输入凭证 |
| **Notion** | 邮箱登录后，自动进入关联 workspace，多个时选择 |
| **GitLab SaaS** | 全局账号，group/project 层面隔离 |
| **钉钉** | 手机号 → 已有组织则进入，无组织则创建新团队 |

**TokenJoy 方案**：与飞书/钉钉一致——凭证验证后，后端查找关联企业，多个时让用户选择。

---

## 3. 流程设计

### 3.1 OTP 登录流程（手机号/邮箱验证码）

```
POST /auth/otp/verify { target, code }
    │
    ▼
验证码正确
    │
    ├── 查找 members 表：哪些企业包含这个 phone/email？
    │
    ├── 0 个匹配
    │       │
    │       ├── registrationEnabled = true → { action: "register", token }
    │       │       → 前端跳转注册页（填公司名 → 创建 Trial）
    │       │
    │       └── registrationEnabled = false → { action: "denied", message: "未找到账号" }
    │
    ├── 1 个匹配
    │       └── → { action: "login", memberId, companyId }
    │           → 后端直接签发 JWT，前端进入系统
    │
    └── 多个匹配
            └── → { action: "select_company", companies: [...] }
                → 前端展示选择页，用户选择后
                → POST /auth/otp/select { target, companyId, sessionToken }
                → 签发 JWT
```

### 3.2 邮箱密码登录流程

```
POST /auth/login { email, password }
    │
    ▼
    ├── 查找 members 表：哪些企业包含这个 email？
    │
    ├── 0 个匹配 → 401 "邮箱或密码错误"
    │
    ├── 1 个匹配 → 验证密码 → 签发 JWT
    │
    └── 多个匹配 → 验证密码（对所有匹配企业）
            │
            ├── 只有 1 个企业密码正确 → 签发 JWT
            └── 多个企业密码正确 → { action: "select_company", companies }
```

### 3.3 邀请加入（Invite）

```
管理员邀请 → 生成邀请码 → 发送给成员
    │
    ▼
成员访问 /invite/accept?code=xxx
    │
    ├── 已登录：从 session 取 userID → AcceptInvite(userID, code, name)
    └── 未登录：设置密码 → 创建/更新 User → AcceptInvite(userID, code, name)
        → 签发 JWT → 进入系统
```

邀请加入时 company_id 在邀请码里已明确，不需要解析。

## 4. 成员查找逻辑

```sql
-- 按手机号查找
SELECT m.id, m.company_id, c.name AS company_name, c.type, c.status
FROM members m
JOIN companies c ON c.id = m.company_id
WHERE m.phone = $1 AND m.status = 'active' AND c.status = 'active';

-- 按邮箱查找
SELECT m.id, m.company_id, c.name AS company_name, c.type, c.status
FROM members m
JOIN companies c ON c.id = m.company_id
WHERE m.email = $1 AND m.status = 'active' AND c.status = 'active';
```

**注意**：只返回 `status = active` 的企业和成员。被冻结的企业不出现在选择列表中。

## 6. API 响应类型

```typescript
type OTPVerifyResult =
  | { action: 'login'; memberId: string; companyId: string }
  | { action: 'select_company'; companies: CompanyBrief[]; sessionToken: string }
  | { action: 'register'; token: string }
  | { action: 'denied'; message: string }

interface CompanyBrief {
  companyId: string
  companyName: string
  companyType: CompanyType
  memberRole: string    // 用户在该企业的角色
}
```

## 7. 私有化简化

私有化模式下，上述所有"查找企业"逻辑被跳过：

```
OTP verify 通过
    └── 直接在 LocalCompanyID 的 members 中查找
        ├── 找到 → 登录
        └── 找不到 → "未找到账号，请联系管理员"
```

不存在多企业选择、不存在注册流程。

---

## 8. 安全考量

| 风险 | 缓解 |
|------|------|
| 枚举攻击（遍历手机号探测企业） | OTP verify 需要先通过验证码，不暴露企业信息 |
| 多企业信息泄露 | 选择页只展示公司名和角色，不暴露其他成员信息 |
| sessionToken 劫持 | select 用的 sessionToken 5 分钟有效 + 一次性 |
| 已冻结企业 | 查找时过滤 `c.status = 'active'`，用户看不到被冻结企业 |

---

## 9. 实施依赖

| 前置 | 内容 |
|------|------|
| OTP 服务 | 手机号/邮箱验证码的 Send/Verify |
| members 联合索引 | `(company_id, phone)` 和 `(company_id, email)` UNIQUE |
| `/auth/otp/verify` 响应扩展 | 返回 action + companies 列表 |
| 前端选择页组件 | `company-select.tsx` |

---

## 10. 与邀请流程的关系

| 流程 | 企业定位方式 | 创建 member 时机 |
|------|-------------|-----------------|
| **自助注册** | 用户创建新企业 → company_id 由注册流程生成 | 注册时创建超管 member |
| **邀请加入** | 邀请码明确指向 company_id | accept-invite 时创建 member |
| **OTP 登录** | 从 members 表反查 company_id | 不创建 member（已存在） |
| **邮箱密码登录** | 从 members 表反查 company_id | 不创建 member（已存在） |

**成员生命周期**：成员只通过两种方式产生——自助注册（创建企业时同时创建超管）和管理员邀请/导入。登录不创建新成员。
