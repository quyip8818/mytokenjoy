## 1. 上线前（P0）

### 1.1 NewAPI / Gateway 联调签字

自动化脚本已就绪；须在真实 full-stack 环境跑通方可视为上线完成：

```bash
# 前提：Backend full-stack .env（见 apps/backend/.env.example）、Backend 已启动
pnpm verify:gate           # 通路冒烟（自建 Key + Gateway + webhook）
pnpm verify:integration    # ledger + Toggle/Rotate/Revoke + metrics（需 NEW_API_ADMIN_TOKEN）
pnpm verify                # lint + test + build
```

| 注意项 | 说明 |
| --- | --- |
| Backend 须先启动 | `verify:gate` 起 newapi compose；Gateway 依赖 Backend `:8080` |
| `NEW_API_ADMIN_TOKEN` | 仅 `verify:integration` 需要 |
| Webhook secret | Backend `NEW_API_WEBHOOK_SECRET` = NewAPI `MANAGEMENT_WEBHOOK_SECRET` |
| 生产契约 | `DEPLOY_ENV=production` 时 NewAPI + Gateway + 入账为启动硬依赖 |

**仍建议人工补验：** SaaS 多企业隔离（跨租户 Gateway / ingest smoke）。

脚本：[apps/newapi/scripts/_verify-lib.sh](../apps/newapi/scripts/_verify-lib.sh) · [gate-verify.sh](../apps/newapi/scripts/gate-verify.sh) · [integration-verify.sh](../apps/newapi/scripts/integration-verify.sh)

### 1.3 Update 严格 Remote-first（按需）

## 2. 后端边界收口（P1）

分层单体方向不变（`Handler → Domain → Store`）；以下为边界收紧，不为架构而架构。

### 2.1 domain 去 `integration/newapi` 依赖

约 **15** 个 domain 文件直接 import `integration/newapi`（`gateway/`、`billing/`、`newapisync/`、`models/`、`company/`、`usage/` 等）。

**目标：** 单位换算等厂商类型留在 integration facade（如 `PointFromNewAPIQuota`）；domain 只依赖 port。`gateway/billing` 先行。

### 2.2 domain 去 `infra/permission` 依赖

泄漏点：`usage/scope.go`、`org/structure/role.go`、`company/service_*.go` 等。

**目标：** 注入 `ScopeResolver` 接口；manifest 解析留在 `infra/permission`。

### 2.3 member 删除去 `identity/httpx` 依赖

`org/structure/member_delete.go` 仍 import `httpx`。

**目标：** handler 传 `currentMemberID`，domain 不感知 HTTP 层。

### 2.4 `LoadBudgetContext` 收敛

`LoadBudgetTreeWithConsumed` 等 snapshot 加载仍有 **10+** 直接调用点（`keys/`、`budget/`、`newapisync/`）。

**目标：** 统一走 `pkg/budget/context.go` 的 `LoadBudgetContext`；Precheck 可进一步收窄为 `BudgetContext` port（参照 `wire_gateway.go`）。

### 2.5 `IsNewAPIUnavailable` 下沉

`newapisync/outbox_errors.go` 依赖 `usage.IsNewAPIUnavailable`。

**目标：** 移到 `pkg/errors` 或 `domain/errors`，断 `newapisync → usage` 边。

### 2.6 worker 测试复用 app wiring

`tests/testutil/worker/runner.go` 手工拼 service，与 `app/` 装配重复。

**目标：** 复用 `app.NewWithStore` 或导出 `buildDomainServices` 测试入口。

## 5. 命名约定（执行时遵守）

与 [Backend-架构.md](./Backend-架构.md) §0 一致。

| 项 | 取值 |
| -- | ---- |
| Gateway 开关 | `NEW_API_GATEWAY_ENABLED` → `GatewayEnabled` |
| SaaS 共享 group | `PLATFORM_SHARED_NEW_API_GROUP` |
| Deps | `Gateway`（类型 `GatewayService`） |
| 平台创建 ProviderKey | `CreateProviderKeyForPlatform` |
| org 本地结构 | `structure.LocalService` |
| Go 缩写 | `ID` / `API` |
| Store 接口 / 实现文件 | `*_repo.go` |

领域词汇：`Gateway` / `NewAPISync` / `PlatformKey` / `ProviderKey` / `NewAPIKey` / `PlatformKeyMapping` / `AsyncJobs`。不用 Relay；领域不用 Token 指 Key。

**不改：** JSON tag / DB 列 / HTTP path（对外契约）；`PlatformKey` → TokenJoyKey；`LogStore` → `LogRepository`。

**接受的双名：** `memberanalytics` ↔ `handler/me`；`LogStore`；`OverrunProcessor` / `Rebalancer`。

## 6. 明确不做

| 不做 | 原因 |
| --- | --- |
| 微服务 / 模块单体拆分 | 团队规模与部署形态不需要 |
| wire、fx 等 DI 框架 | 构造函数注入已够用 |
| `domain` → `bounded` / `http` → `transport` | 纯改名 |
| `domain/types` 回迁各域 | 单一 DTO 层更实用 |
| 全量 service 改窄 repo 注入 | 按场景渐进 |
| 内存 store mock | PG per-schema 已是标准 |
| scaffold 代码生成 | 域新增频率低 |
| 5 个独立 queue repository | 底层一张 `async_jobs` 表 |
| audit/platform handler 大规模重构 | 顺手对齐即可 |
