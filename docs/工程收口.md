## 1. 上线前（P0）

### 1.1 NewAPI / Gateway 联调签字

自动化脚本已就绪；须在真实 full-stack 环境跑通方可视为上线完成。

```bash
# 前提：apps/backend/.env（见 .env.example Full-stack 块）、Backend 已监听 :8080
pnpm verify:gate           # 通路：自建 Key + Gateway + webhook
pnpm verify:integration    # 完整栈：ledger + Toggle/Rotate/Revoke + metrics（需 NEW_API_ADMIN_TOKEN）
pnpm verify                # lint + test + build
```

| 项 | 说明 |
| --- | --- |
| Backend 须先启动 | `verify:gate` 会起 newapi compose；Gateway 反代依赖 Backend |
| `NEW_API_ADMIN_TOKEN` | 仅 `verify:integration` 需要 |
| Webhook secret | Backend `NEW_API_WEBHOOK_SECRET` = NewAPI `MANAGEMENT_WEBHOOK_SECRET` |
| 生产契约 | `DEPLOY_ENV=production` 时 NewAPI + Gateway + 入账为启动硬依赖 |

**人工补验（脚本不覆盖）：** SaaS 多企业隔离 — 跨租户 Gateway / ingest smoke。

脚本：[`_verify-lib.sh`](../apps/newapi/scripts/_verify-lib.sh) · [`gate-verify.sh`](../apps/newapi/scripts/gate-verify.sh) · [`integration-verify.sh`](../apps/newapi/scripts/integration-verify.sh)

### 1.2 通知失败可观测

`infra/notification/service.go`：`NOTIFY_WEBHOOK_URL` HTTP 失败仅写 log，`Send()` 仍 `return nil`；调用方无法感知、无法重试。

### 1.3 Update Remote-first（按需）

`keys/platform_key_update.go`：配额 / 白名单更新为先写 DB 再 `SyncUpdatePlatformKey`、失败回滚；进程崩溃窗口仍可能短暂不一致。若上线要求与 Create 路径铁律一致，改为先 Remote。

## 4. 约束

**命名与分层** — 执行时遵守 [Backend-架构.md](./Backend-架构.md) §0（Gateway / NewAPISync / PlatformKey 词汇、配置项、分层边界）。

**明确不做**

| 不做 | 原因 |
| --- | --- |
| 微服务 / 模块单体拆分 | 团队与部署形态不需要 |
| wire、fx 等 DI 框架 | 构造函数注入已够用 |
| `domain` → `bounded` / `http` → `transport` | 纯改名 |
| `domain/types` 回迁各域 | 单一 DTO 层更实用 |
| 全量 service 改窄 repo 注入 | 按场景渐进 |
| 内存 store mock | PG per-schema 已是标准 |
| scaffold 代码生成 | 域新增频率低 |
| 5 个独立 queue repository | 已收口为 River `river_job`（[实现-离线任务管理.md](./实现-离线任务管理.md)） |
| audit/platform handler 大规模重构 | 顺手对齐即可 |
