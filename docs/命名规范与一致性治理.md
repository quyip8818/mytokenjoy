# 命名规范与一致性治理

> 状态：正式规范（架构师制定）
> 版本：v1.0
> 日期：2026-07-24
> 目标：建立清晰、一致、易维护的命名体系，防止未来漂移

---

## 一、核心理念

### 1.1 单一语义原则
每个业务领域名词在整个系统中保持**单一且明确的语义**：

| 名词 | 定义 | 使用范围 |
|------|------|----------|
| **member** | 组织成员（第三人称） | 组织管理上下文（org domain） |
| **me** | 当前用户（第一人称） | 个人管理上下文（my domain） |
| **notification** | 系统通知 | 消息推送、通知管理 |
| **budget** | 预算管理 | 资源分配、费用控制 |
| **billing** | 计费系统 | 钱包、充值、发票 |
| **wallet** | 钱包（产品术语） | UI展示层，面向用户 |

### 1.2 一致性层级
从底层到展示层，保持命名对齐：
```
领域语义 → 后端架构 → API接口 → 前端逻辑 → 前端路由
```

### 1.3 命名风格规则

| 层级 | 命名风格 | 示例 |
|------|----------|------|
| Go 包名 | 全小写单词，单数 | `notification`, `approval` |
| Go 文件名 | snake_case | `platform_key_create.go` |
| API 路径前缀 | 领域名原样 | `/budget`, `/org`, `/me` |
| 前端 feature 目录 | kebab-case，领域概念 | `features/budget/` |
| 前端 API 文件 | kebab-case，与 feature 对齐 | `api/budget.ts` |
| 前端 API 导出 | `{domain}Api` camelCase | `budgetApi`, `notificationApi` |
| 前端 route 目录 | kebab-case，与 URL 对齐 | `routes/budget/` 对应 `/budget` |
| 前端 query-key | feature 领域名 | `['budget', ...]` |
| 权限 key | `{domain}:{action}` | `budget:read`, `org:members` |
| 前端权限常量 | `UPPER_SNAKE_CASE` | `BUDGET_READ` |

## 二、现状分析与问题诊断

### 2.1 当前命名问题（按风险排序）

| 风险等级 | 问题 | 示例 | 影响范围 | 优先级 |
|----------|------|------|----------|--------|
| **P0 高风险** | member/me 语义混淆 | `features/member/` 存放"我的看板"（应是 me） | 整个用户个人功能 | 立即 |
| **P1 中风险** | 目录与 URL 不对齐 | `routes/member/` 对应 `/me/*` URL | 路由配置，开发者困惑 | 近期 |
| **P1 中风险** | API 文件与 feature 不对齐 | `api/notification.ts` vs `features/notifications/` | 导入路径不一致 | 近期 |
| **P1 中风险** | 前后端领域名分裂 | 后端 `billing`，前端 `wallet` | 跨层协作困难 | 近期 |
| **P2 低风险** | 单复数不统一 | `/notifications`（复数）vs `/budget`（单数） | 风格不一致 | 可选 |

### 2.2 语义混淆详细分析

#### 2.2.1 member 一词多义
- **作为名词**：`Member`（组织成员） - `api/types/org.ts`
- **作为形容词**：`MemberDashboardView`（我的看板） - `api/types/member.ts`
- **作为目录**：`features/member/`（实际是"我的看板"）

#### 2.2.2 URL 与目录不对齐
- **URL 路径**：`/me/*`（第一人称，当前用户）
- **目录名称**：`routes/member/`（第三人称，组织成员语义）

#### 2.2.3 计费领域分裂
- **后端全链路**：`domain/billing` → `handler/billing` → `/api/billing/*`
- **前端领域**：`features/wallet/`，`routes/wallet/`
- **产品语言**："钱包管理"（UI标签）

## 三、统一命名规范

### 3.1 完整领域映射表（目标状态）

| 领域 | 后端 domain | API 前缀 | 前端 feature | 前端 route | URL 路径 | 权限前缀 | 状态 |
|------|------------|----------|-------------|------------|----------|----------|------|
| 组织 | org | `/org` | org | routes/org | `/org/*` | `org:` | ✅ |
| 预算 | budget | `/budget` | budget | routes/budget | `/budget` | `budget:` | ✅ |
| 密钥 | keys | `/keys` | keys | routes/keys | `/keys/*` | `keys:` | ✅ |
| 模型 | models | `/models` | models | routes/models | `/models/*` | `model:` | ✅ |
| 仪表盘 | dashboard | `/dashboard` | dashboard | routes/dashboard | `/dashboard/*` | `dashboard:` | ✅ |
| 审计 | audit | `/audit` | audit | routes/audit | `/audit/*` | `audit:` | ✅ |
| **当前用户** | me | `/me` | mydashboard + account | routes/me | `/me/*` | `self:` | ✅ |
| 通知 | notification | `/notifications` | notifications | (内嵌) | - | - | ✅ |
| 审批 | approval | `/approvals` | approval | routes/approval | `/approvals` | - | ✅ |
| **计费系统** | billing | `/billing` | billing | routes/billing | `/wallet` | `billing:` | ✅ |
| 会话 | session | `/session` | session | - | - | - | ✅ |
| 试用 | trial | - | trial | - | - | - | ✅ |

**状态说明**：
- ✅：命名一致，无需改动
- ⚠️：存在命名不一致，建议修正

### 3.2 具体命名规则

#### 3.2.1 后端（Go）
```go
// 领域包（小写单数）
domain/budget/
domain/notification/

// Handler包（小写单数）  
handler/budget/
handler/notification/

// 文件名（snake_case）
budget_tree.go
member_budget.go

// 路由注册
r.Route("/budget", ...)      // 领域命名空间（单数）
r.Route("/notifications", ...) // 集合资源（复数，REST惯例）
```

#### 3.2.2 前端（TypeScript/React）
```typescript
// API文件（与feature目录对齐）
api/budget.ts          // ✅ 正确：与 features/budget/ 对齐
api/notifications.ts   // ⚠️ 应改为：与 features/notifications/ 对齐

// Feature目录（领域概念）
features/budget/
features/me/           // ⚠️ 应改为：替代 features/member/

// Route目录（与URL对齐）
routes/budget/         // ✅ 正确：对应 /budget
routes/me/             // ⚠️ 应改为：替代 routes/member/，对应 /me/*

// Query keys
const budgetKeys = { all: ['budget'] }
const meKeys = { all: ['me'] }  // ⚠️ 应改为：替代 memberKeys
```

### 3.3 特殊情况处理

#### 3.3.1 产品名与系统名不同（billing/wallet）
**处理原则**：
1. 后端统一使用系统名：`billing`
2. 前端领域逻辑使用系统名：`features/billing/`
3. 前端路由路径可用产品名：`/wallet`（需权衡）
4. UI标签使用产品名："钱包管理"

**选项评估**：
- **选项A**：统一为 `billing`（技术一致性好）
- **选项B**：保持 `wallet`（产品体验好）
- **建议**：评估业务扩展性，如果未来扩展发票、订阅等，建议统一

#### 3.3.2 单复数约定
**规则**：
- **领域命名空间**用单数：`/budget`, `/org`, `/me`
- **集合资源**可用复数：`/notifications`, `/approvals`
- **特殊情况**：`keys`, `models` 作为领域名保持复数形式

**理由**：
- 大多数API路由是领域命名空间，非集合资源
- `/notifications`, `/approvals` 是集合语义，用复数合理
- 保持现有约定，不改动外部API契约

## 五、防漂移治理机制

### 5.1 新增领域检查清单
**新增业务领域时，必须检查以下项：**

#### 5.1.1 语义确认
- [ ] 这是新领域还是现有领域的子功能？
- [ ] 领域名词是否有明确、单一的定义？
- [ ] 是否与现有语义冲突？（特别是 member/me）

#### 5.1.2 后端命名
- [ ] Go domain包名：`domain/{name}`（小写单数）
- [ ] Go handler包名：`handler/{name}`（小写单数）
- [ ] API路由前缀确定：领域命名空间（单数）或集合资源（复数）

#### 5.1.3 前端命名
- [ ] API文件：`api/{name}.ts`（必须与feature目录一致）
- [ ] Feature目录：`features/{name}/`
- [ ] Route目录：`routes/{url-prefix}/`（必须与URL对齐）
- [ ] 导出名：`{name}Api` camelCase
- [ ] Query keys：`{name}Keys`

#### 5.1.4 权限命名
- [ ] 权限key：`{domain}:{action}`
- [ ] 前端常量：`{DOMAIN}_{ACTION}`（大写蛇形）

### 5.2 代码审查强制检查点
**代码审查时，必须验证以下一致性：**

1. **目录与URL对齐**：`routes/{x}/` 是否对应 `/x/*` URL？
2. **API与feature对齐**：`api/{x}.ts` 是否对应 `features/{x}/`？
3. **语义正确性**：member是否被错误用于me的语义？
4. **新增领域合规**：是否遵循检查清单？

### 5.3 定期审计机制
**每季度执行一次命名一致性审计：**

1. **文件扫描**：
   - 检查所有 `features/` 目录与对应 `api/` 文件
   - 检查所有 `routes/` 目录与 `config/routes.ts`

2. **语义验证**：
   - 验证 member/me 语义分离情况
   - 检查 billing/wallet 一致性

3. **文档更新**：
   - 更新本文档中的映射表
   - 记录新发现的不一致点

4. **改进建议**：
   - 提出下一季度命名优化建议
   - 评估自动化检查工具需求

## 七、决策记录与权衡

### 7.1 关键决策
1. **member/me 彻底分离**（采纳）
   - 理由：语义混淆是最高风险
   - 权衡：重构成本 vs 长期可维护性

2. **分阶段实施策略**（采纳）
   - 理由：风险可控，团队可逐步适应
   - 权衡：彻底重构 vs 渐进改进

3. **接受单复数差异**（采纳）
   - 理由：外部API契约，改动成本高
   - 权衡：完全一致 vs 向后兼容

4. **billing/wallet 统一评估**（待定）
   - 理由：需评估业务扩展需求
   - 权衡：技术一致性 vs 产品体验

### 7.2 实施优先级矩阵
| 优先级 | 标准 | 包含项目 |
|--------|------|----------|
| **P1** | 无业务逻辑变更，明显错误 | notification.ts 对齐，routes/member 对齐 |
| **P2** | 有业务影响但语义重要 | member→me 语义分离 |
| **P3** | 权衡性改进，可选 | billing/wallet 统一 |
