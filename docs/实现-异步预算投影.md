# 实现：异步预算投影

> **范围**：未上线；`schema.sql` **破坏性更新** + wipe，无历史数据、**无向后兼容**。  
> **模式**：ledger 单写、consumed 异步、Gateway 零 consumed JOIN、Overrun 最终执法。  
> **前置（已交付）**：River 离线任务 + **定时任务**（`river.Client` Periodic、`jobs.Enqueuer`）；见 [Backend-离线任务.md](./Backend-离线任务.md)。本文聚焦 **投影域** 与 Gateway 瘦化；fanout kind 注册见 §12.1 / §8。

---

## 1. 路径对照

| 路径 | 现网 | 终态 |
| --- | --- | --- |
| **Ingest** | `enforceBudgetCap` → lot → ledger → `Apply` → rebalance + Overrun | lot → ledger → `InsertTx(budget_project)` + `InsertTx(wallet_sync)` |
| **Projector** | — | Increment → `budget_consumed`；进度 → `budget_projection_progress`；批末 `Insert(rebalance)` + `Insert(overrun)` |
| **Reconcile** | — | `ExpectedConsumed` ↔ `budget_consumed`；`SetConsumed`；Periodic 扇出 `budget_reconcile`；漂移后 `Insert(rebalance)` |
| **Dashboard 投影** | Ingest 同步 `UpsertBucket` | **每小时**增量 → `usage_buckets`；**每天**全量 reconcile 修漂移（§12；**River Periodic fanout** 注册，见 §12.1） |
| **Gateway `/v1`** | CTE + limit JOIN + 4× `budget_snapshots` | `platform_keys` + `companies` + allowlist（**0 consumed JOIN**） |
| **软挡单** | — | 可选 Redis `GatewayBudgetCheck`（**不进 PG SQL**） |

**产品差异（须书面备案）：**

| 项 | 现网 / PRD | 终态 |
| --- | --- | --- |
| PRD US-08「100% 阻断请求」 | Gateway 预检硬拦（`checkBudgetRemain`） | Gateway **best-effort**（钱包 + 可选 Redis 软挡）；组织预算靠 Overrun ≤1s Disable |
| 超卖窗口 | Precheck 通过后、入账前（短） | Gateway 放行 → ledger → Projector → Overrun（**更长**；高并发边界 Key 可能连续多笔） |
| 执行面门禁 | Precheck + 同事务 snapshots + Rebalance | 钱包（Gateway）+ NewAPI `remain_quota`（**Projector 后 Rebalance**）+ Overrun Disable |

不变：`ComputeRemainBudget`（Overrun / 软挡 / 控制台 / Rebalance）、挂组跳过 member、lot FIFO、River `wallet_sync` job、`AcquireBudgetLock`。

## 2. `/v1` 性能约束

| 指标 | 要求 |
| --- | --- |
| Gateway store 调用 | **≤ 1**（`LoadPrecheckContext`） |
| PG 预检 | `platform_keys` ⋈ `companies` + allowlist 子查询 |
| 禁止 JOIN | `budget_consumed`、`budget_snapshots`、`usage_ledger`；**禁止**为 consumed 读 `members` / `org_nodes` / `budget_groups` |
| 预检 HTTP | **0** |
| `Evaluate` | 纯内存：wallet + key_status + allowlist（+ 可选 Redis `soft_remain`） |
| 软挡 | 最多 **+1 Redis GET**；miss 降级放行，**禁止** miss 回源 PG |

瘦化目标：去掉现网 `gateway_precheck_repo` 里 4 路 `budget_snapshots` LEFT JOIN（见现网 L100–119），**不新增**任何 PG I/O。

**与 `架构简化方案.md` Phase 5 的差异：** 本方案 Redis 仅承载可选 `GatewayBudgetCheck`（soft remain），**miss 不回源 PG**；Phase 5 的 `PrecheckCache` 全量上下文 + miss fallback Postgres **不在本方案范围**。

```mermaid

  ORQ --> OV[Overrun → Disable Key]
  BD --> RB
  BD --> OV

  SCH[River Periodic] --> REC[budget_reconcile fanout]
  UL --> REC
  REC --> BD
  REC -.->|ExpectedConsumed 共用归因| BP
  REC -.->|漂移修复后| RBQ

  SCH --> DP[dashboard_project fanout 每小时]
  UL --> DP
  DP --> UB[(usage_buckets)]

  GW -.->|可选| GBC
```

**删表：** `budget_snapshots`。  
**保留表：** `usage_buckets`（纯看板投影，§12；Ingest 不再写）。  
**新增 PG：** `budget_consumed`、`budget_projection_progress`、`dashboard_projection_progress`（§12.2）。

**Rebalance 时序约束（关键）：** Rebalance 读 `budget_consumed` 算 `ComputeRemainBudget` → NewAPI `remain_quota`。consumed 异步后，**禁止** Ingest 入队 Rebalance（会读到 stale consumed，执行面 quota 系统性偏高）。Rebalance **仅**在 Projector 批提交后（及 Reconcile 修漂移后）入队。

## 4. Schema

### 4.1 `budget_consumed`（四轴 consumed，唯一投影表）

```sql
CREATE TABLE budget_consumed (
    company_id  BIGINT NOT NULL REFERENCES companies (id) ON DELETE CASCADE,
    axis_kind   TEXT NOT NULL CHECK (axis_kind IN (
        'platform_key', 'member', 'org_node', 'budget_group'
    )),
    axis_id     TEXT NOT NULL,
    period_key  TEXT NOT NULL,
    consumed    NUMERIC(18, 6) NOT NULL DEFAULT 0,
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (company_id, axis_kind, axis_id, period_key)
);

CREATE INDEX idx_budget_consumed_overrun
    ON budget_consumed (company_id, axis_kind, period_key);
```

| axis_kind | axis_id |
| --- | --- |
| `platform_key` | `platform_keys.id` |
| `member` | `members.id` |
| `org_node` | `org_nodes.id`（含祖先 rollup） |
| `budget_group` | `budget_groups.id` |

**`/v1` 不读。** 替代整张 `budget_snapshots`。

### 4.2 `budget_projection_progress`（投影进度，独立表）

```sql
CREATE TABLE budget_projection_progress (
    company_id            BIGINT NOT NULL REFERENCES companies (id) ON DELETE CASCADE,
    stream                TEXT NOT NULL DEFAULT 'ledger_consumed',
    last_occurred_at      TIMESTAMPTZ,
    last_ledger_id        TEXT,
    updated_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (company_id, stream)
);
```

| 列 | 含义 |
| --- | --- |
| `stream` | 投影流名称；现仅 `ledger_consumed`，预留扩展 |
| `last_occurred_at` | 已投影到的最后一行 ledger 的 `occurred_at` |
| `last_ledger_id` | 已投影到的最后一行 ledger 的 `id` |

二者合用排序 `(occurred_at ASC, id ASC)`，避免同一时刻多行漏算/双算。  
首跑两列为 NULL；**整批 Increment + 进度更新同一事务**；批失败不推进。

**`/v1` 不读。** 不挂 `companies`，避免污染租户主表、方便多 stream 扩展。

### 4.3 Projector 读 ledger 索引

增量查询：`WHERE company_id = $1 AND (occurred_at, id) > ($last_occurred_at, $last_ledger_id) ORDER BY occurred_at ASC, id ASC LIMIT $batch`。

建议追加（或替换仅 DESC 的索引）：

```sql
CREATE INDEX idx_usage_ledger_projector_cursor
    ON usage_ledger (company_id, occurred_at ASC, id ASC)
    WHERE event_type = 'call_settled';
```

### 4.4 并发

- `budget_project` job：River Unique（`ByState` **不含 completed**），见 [Backend-River实现.md](./Backend-River实现.md) §4。
- Projector 事务内：`SELECT ... FROM budget_projection_progress WHERE company_id = $1 FOR UPDATE`，再读 ledger、写 `budget_consumed`、推进 progress。
- **批大小上限**：单批建议 ≤500～1000 行 ledger；超出则本批 commit 后 `Insert(budget_project)` 续跑，避免长事务。
- **可观测**：暴露 `projection_lag_seconds = now() - last_occurred_at`（按 company），支撑 §15 验收 #6。

## 7. BudgetProjector

### 7.1 触发

| 字段 | 值 |
| --- | --- |
| kind | `budget_project` |
| 入队 | Ingest `InsertTx`；批末 `Insert` 续跑 |
| throttle | `UniqueOpts{ByArgs, ByPeriod: 1s, ByState: 不含 completed}` |
| 详见 | [Backend-River实现.md](./Backend-River实现.md) §4 |

### 7.2 单批事务

```text
1. FOR UPDATE budget_projection_progress(company_id, 'ledger_consumed')
2. SELECT ledger WHERE (occurred_at, id) > (last_occurred_at, last_ledger_id)
   ORDER BY occurred_at ASC, id ASC LIMIT $batch_max
3. FOR each entry: ApplyIncrement（内部 ConsumptionDeltas → +=）→ budget_consumed
4. UPDATE budget_projection_progress ← 本批最后一行
5. COMMIT
6. `Insert(rebalance)`（本批触及轴）  -- 事务外；consumed 已提交
7. `Insert(overrun)`                                               -- 事务外
8. 可选 SET GatewayBudgetCheck                                     -- 事务外
9. 若本批行数 = batch_max：`Insert(budget_project)` 续跑（Unique 不含 completed，[Backend-River实现.md](./Backend-River实现.md) §4）
```

**Rebalance 入队轴：** 与现网 `side_effects.go` 相同规则——有 `member_id` 则 member 轴；每次 department 轴；有 `budget_group_id` 则 group 轴。**禁止**在步骤 5 之前入队。

### 7.3 共用归因层（Projector / Reconcile 共享）

**原则：** Projector 与 Reconcile **共用 consumed 归因逻辑**，但**不共用写库语义**。

| 函数 | 职责 | 调用方 |
| --- | --- | --- |
| `ConsumptionDeltas(entry, open)` | 纯函数：一行 ledger → `[]AxisDelta`（含 org 祖先展开） | Projector、Reconcile |
| `ApplyIncrement(ctx, writer, entry, open)` | 对 writer 做 `+=`（含 `RollupOrgNodeAncestors`） | **仅 Projector** |
| `ExpectedConsumed(ctx, entries, clk)` | 遍历 ledger，`SUM(ConsumptionDeltas)` → `map[AxisKey]float64` | **仅 Reconcile** |

```go
type AxisDelta struct {
    Kind      AxisKind   // platform_key | member | org_node | budget_group
    AxisID    string
    PeriodKey string     // OpenDepartmentPeriod；≠ ledger.period_key（发生轨）
    Amount    float64
}

// 自现网 projection.go 抽出；domain/budget/consumed_attrib.go
func ConsumptionDeltas(ctx, entry, open OpenBudgetPeriod) ([]AxisDelta, error)

func ApplyIncrement(ctx, writer ConsumedIncrementWriter, entry, open) error {
    deltas, err := ConsumptionDeltas(ctx, entry, open)
    // writer.IncrementConsumed / RollupOrgNodeAncestors
}

func ExpectedConsumed(ctx, entries []LedgerEntry, clk) (map[AxisKey]float64, error) {
    acc := map[AxisKey]float64{}
    for _, e := range entries {
        open, _ := OpenDepartmentPeriod(ctx, nodes, e.DepartmentID, clk)
        for _, d := range ConsumptionDeltas(ctx, e, open) {
            acc[AxisKey{d.Kind, d.AxisID, d.PeriodKey}] += d.Amount
        }
    }
    return acc, nil
}
```

**`ConsumptionDeltas` 规则（与现网 `projection.go` 一致）：**

```text
open = OpenDepartmentPeriod(dept, business_clock)   -- 开账轨；禁止用 ledger.period_key / OccurredAt

platform_key_id           → AxisDelta(platform_key)
budget_group_id?          → AxisDelta(budget_group)
member_id?                → AxisDelta(member)
department_id             → 每个祖先 org_node 一条 AxisDelta（与 RollupOrgNodeAncestors 等价）
```

**禁止：**

- Reconcile 修复漂移时调用 `ApplyIncrement` 做 `+=`（会与 Projector 重复累加）
- Reconcile 单独写 SQL `GROUP BY` + 另一套 org rollup（易与 OpenPeriod 规则分叉）
- Reconcile 全量 replay `ApplyIncrement` 到临时表（正确但大 tenant 成本高；优先 `ExpectedConsumed` 内存聚合）

### 7.4 Projector 写路径

```text
open = OpenDepartmentPeriod(entry.DepartmentID, business_clock)
ApplyIncrement(ctx, budgetConsumedRepo, entry, open)   -- 内部 ConsumptionDeltas → +=
```

**不含** `usage_buckets` Upsert（看板见 §12）。

## 8. Reconcile

| 项 | 值 |
| --- | --- |
| channel | per-company：`budget_reconcile`；Periodic 扇出：`budget_reconcile_fanout`（定案 kind 名以实现为准） |
| 频率 | 每 15～30min 覆盖全 tenant（**River Periodic** → fanout job → 批量 `Insert` 子 job） |
| 队列 | `low`（见 [Backend-River实现.md](./Backend-River实现.md) §5） |
| 读 ledger | 当前开账窗口内全量（或按 company 分页）；**不用** ledger.period_key 当 consumed 账期 |
| 期望 consumed | `ExpectedConsumed(ctx, ledgerRows, clk)` — **必须**走 §7.3 共用归因 |
| 对比 | `ExpectedConsumed` ↔ `budget_consumed`（按 `AxisKey`） |
| 修复 | 漂移行 **`SetConsumed`**（绝对值）；**禁止** `ApplyIncrement` |
| 副作用 | 告警 + 可选刷新 GatewayBudgetCheck；漂移修复后 **EnqueueRebalance** |

```text
1. SELECT ledger（company, 开账相关窗口）
2. expected := ExpectedConsumed(entries, business_clock)
3. actual   := budget_consumed 全量 / 按 period 拉取
4. FOR each AxisKey WHERE expected ≠ actual（容差 ε）:
     SetConsumed(kind, id, period, expected)
     告警
5. EnqueueRebalance（company 或触及轴）
```

**调度（待落地，机制已就绪）：**

```text
River Periodic（如 30min）
  → Insert 1× fanout job（low 队列）
  → budget.FanoutReconcileJobs：ForEachActiveCompany → Insert(budget_reconcile) per tenant
  → budget_reconcile Worker → §8 步骤 1–5
```

注册位置：`internal/infra/river/periodic.go`（与已落地的 `org_sync`、`monthly_rebalance` 同模式）；fanout domain 方法见 [实现-离线任务管理.md §2](./实现-离线任务管理.md#2-periodic-fanoutstep-5-剩余)。

**不进 `/v1`。**

## 9. Gateway `/v1`

### 9.1 终态 PG SQL（默认档）

去掉现网 `base` CTE 中 mapping / members / org_nodes / budget_groups / budget_snapshots，保留：

```sql
SELECT
    c.id AS company_id,
    c.status AS company_status,
    c.balance_point,
    c.newapi_wallet_user_id,
    pk.id AS platform_key_id,
    pk.status AS key_status,
    pk.key_hash,
    EXISTS (
        SELECT 1 FROM model_allowlist ma
        WHERE ma.company_id = pk.company_id
          AND ma.owner_type = 'platform_key'
          AND ma.owner_id = pk.id
    ) AS has_allowlist,
    COALESCE((
        SELECT array_agg(DISTINCT mdl.type ORDER BY mdl.type)
        FROM model_allowlist ma
        JOIN models mdl ON mdl.model_id = ma.model_id
        WHERE ma.company_id = pk.company_id
          AND ma.owner_type = 'platform_key'
          AND ma.owner_id = pk.id
          AND mdl.enabled = TRUE
    ), '{}') AS allowlist_types
FROM platform_keys pk
JOIN companies c ON c.id = pk.company_id
WHERE pk.key_hash = $1;
```

**相对现网行为变化（须测试覆盖）：**

| 变化 | 现网 | 终态 |
| --- | --- | --- |
| `platform_key_mappings` INNER JOIN | 无 mapping 则预检 not found | 仅查 `platform_keys.key_hash`；有 Key 无 mapping 也可预检通过 |
| `dept_found` / 四轴 limit | 同步硬拦 | 不查；配置错误靠控制台 + Overrun |
| 时钟参数 `$2`（period_key） | 用于 snapshot JOIN | 删除；预检不再依赖业务时钟 |

### 9.2 `PrecheckContext` 终态

```text
PrecheckContext {
  Wallet:  { CompanyStatus, BalancePoint, ... }   // 保留
  Routing: { KeyStatus, HasAllowlist, AllowlistTypes, PlatformKeyID }  // 保留

```

### 9.3 `Evaluate` 终态

```text
1. company active
2. balance_point >= minEstimate
3. key_status == active
4. allowlist（若启用）
5.（可选）GatewayBudgetCheck.soft_remain >= minEstimate
```

**不再检查：** 部门预算是否存在、四轴 remain、dept_found。配置错误由 Overrun / 控制台暴露，**不以 `/v1` 多 JOIN 为代价**。

## 11. Overrun

**入队：** 仅 Projector（`budget_consumed` 提交后）。  
**读：** `budget_consumed` + limit 配置；`AcquireBudgetLock` 保留。

### 11.1 检查顺序

| # | 轴 | consumed | limit | Disable |
| --- | --- | --- | --- | --- |
| 1 | platform_key | `budget_consumed(platform_key, pk_id)` | `platform_keys.budget` | **仅该 Key** |
| 2 | member | `budget_consumed(member, id)` | `personal_budget` | 成员非挂组 Keys |
| 3 | department | `budget_consumed(org_node, dept_id)` | `org_nodes.budget` | 部门下 Keys |
| 4 | budget_group | `budget_consumed(budget_group, id)` | `budget_groups.budget` | 组内 Key |

挂组 Key：**跳过** member 轴（与现网 Rebalance / 预检待修复项一致）。  
实现：按上表顺序检查，**首个命中即 Disable 并返回**（重写现网 overrun early-return）。

### 11.2 相对现网变更

| 现网 | 终态 |
| --- | --- |
| 不查 Key 轴 | **新增** platform_key 检查 |
| member 用 `SumMemberKeyUsed` → snapshots | 改 `budget_consumed(member)` |
| dept/group 读 `budget_snapshots` | 改 `budget_consumed` |
| Ingest 入队 Overrun | **删除**；仅 Projector 入队 |

### 11.3 时序

```text
T0  /v1 放行（wallet + 可选软挡）
T1  ledger 入账（lot / balance_point 同步扣减）
T2  +≤1s  Projector → budget_consumed 更新
T2'       EnqueueRebalance → NewAPI remain_quota 对齐 consumed
T3        Overrun → key inactive（若超限）
```

## 12. 控制台与看板

| 场景 | 读法 |
| --- | --- |
| 预算树 | `budget_consumed(org_node)` + limit |
| 成员 / 组 / Key 已用 | `budget_consumed` |
| 看板 hour/day 趋势 | **`usage_buckets`（纯看板投影，§12.1）** |
| 看板 minute | `usage_ledger`（与现网一致） |

### 12.1 定案：保留 `usage_buckets`（纯看板投影）

**不**用 `budget_consumed` 顶替看板——前者是账期轴 consumed，后者是 **时间桶 × 部门 × 成员 × 模型**，语义不同。

| 项 | 约定 |
| --- | --- |
| 写路径 | Ingest **不再** `UpsertBucket`；与 BudgetProjector **独立 job** |
| 桶粒度 | hour（`occurred_at` truncate hour） |
| 可接受 lag | **≤1h**（管理端趋势，非 Gateway 热路径） |
| 增量 | **每小时** cursor 增量投影 |
| 全量 | **每天** reconcile 兜底修漂移 |

| 任务 | channel | 频率 | 机制 |
| --- | --- | --- | --- |
| **Dashboard 增量** | `dashboard_project` | **每小时** | Periodic fanout → 每 tenant 一条增量 job；读 `dashboard_projection_progress` cursor 后 ledger → `UpsertBucket +=` |
| **Dashboard 全量** | `dashboard_reconcile` | **每天** | Periodic fanout → 每 tenant 一条对账 job；ledger `GROUP BY` 期望桶 ↔ `usage_buckets`；漂移 **`SetBucket` 绝对值** |

**调度承载（已定案，待注册 kind）：**

| 能力 | 状态 |
| --- | --- |
| River `PeriodicJob` + leader 选举 | **已落地**（`org_sync`、`monthly_rebalance`） |
| Periodic **fanout** → 全 tenant 子 job | **待落地**（本方案三个 fanout） |
| cron 表达式（日级 reconcile） | River 支持；注册于 `periodic.go` |

实现步骤：domain 实现 `FanoutProjectJobs` / `FanoutReconcileJobs` → `workers/*_fanout.go` 薄壳 → `BuildPeriodicJobs` 追加 interval/cron。详见 [Backend-离线任务.md §6](./Backend-离线任务.md#6-periodic-任务)、[实现-离线任务管理.md §2](./实现-离线任务管理.md#2-periodic-fanoutstep-5-剩余)。

**与 BudgetProjector 解耦：**

| | BudgetProjector | Dashboard 投影 |
| --- | --- | --- |
| 目标表 | `budget_consumed` | `usage_buckets` |
| 目标 lag | ≤1s | ≤1h |
| 触发 | Ingest 入队 `budget_project`（debounce） | **定时**每小时（不经 Ingest 热路径） |
| 归因 | `ConsumptionDeltas` + OpenPeriod | 按 `occurred_at` + ledger 维度直聚合 |

### 12.2 `dashboard_projection_progress`

与 `budget_projection_progress` 同模式，独立 stream：

```sql
CREATE TABLE dashboard_projection_progress (
    company_id            BIGINT NOT NULL REFERENCES companies (id) ON DELETE CASCADE,
    stream                TEXT NOT NULL DEFAULT 'dashboard_buckets',
    last_occurred_at      TIMESTAMPTZ,
    last_ledger_id        TEXT,
    updated_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (company_id, stream)
);
```

增量查询复用 §4.3 `idx_usage_ledger_projector_cursor`（按 `occurred_at, id` 排序）。

### 12.3 Dashboard 增量批事务（`dashboard_project`）

```text
1. FOR UPDATE dashboard_projection_progress(company_id, 'dashboard_buckets')
2. SELECT ledger WHERE (occurred_at, id) > cursor ORDER BY occurred_at ASC, id ASC LIMIT $batch
3. FOR each entry: UpsertBucket（hour 桶 += cost / tokens / call_count）
4. UPDATE dashboard_projection_progress ← 本批最后一行
5. COMMIT
```

**禁止**与 `budget_project` 同事务；**禁止**在 BudgetProjector 内顺带写 buckets。

### 12.4 Dashboard 全量对账（`dashboard_reconcile`）

```text
1. SELECT ledger（company，保留窗口，如近 90 天 call_settled）
2. expected := GROUP BY date_trunc('hour', occurred_at), department_id, member_id, model
3. actual   := usage_buckets 同窗口
4. FOR each bucket key WHERE expected ≠ actual（容差 ε）:
     SetBucket(absolute expected)
     告警
```

全量窗口与保留策略可在实现时配置；首版建议 **90 天** 或 **当前自然月 + 上月**。

US-08 阈值通知（80%/90%）：在 **Projector 批末或 Reconcile** 读 `budget_consumed` + limit 触发，**不在 `/v1`**（与 `alert_rules` 运行时 Worker 一并落地，单独估工时）。
